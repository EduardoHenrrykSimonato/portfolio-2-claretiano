# Histórico de Alterações (Changelog)

Mantemos aqui um log de alto nível para as inteligências artificiais e desenvolvedores rastrearem a evolução do código.

* **Fase 1 (Atual):** 
  * Projeto inicializado utilizando Angular 12 e Ionic 5 para garantir compatibilidade com o ambiente local (Node 12).
  * Criação dos arquivos críticos do workspace (`angular.json`, `package.json`, `tsconfig`).
  * Geração manual dos diretórios `src/app/`, arquivos de modelo e service para atender aos requisitos de dados estritamente em memória.
  * Criação da pasta `/docs` contemplando o escopo, arquitetura, restrições tecnológicas (ex.: proibido Firebase/SQLite) e modelo de dados.
  * Definição e documentação do padrão visual do aplicativo, estabelecendo regras de UI/UX e uso obrigatório de componentes Ionic (`02-padrao-visual.md`).
  * Recriação das interfaces `Chamado` e `Tecnico` com campos exatos exigidos e unificação do estado na memória através do `ChamadosService`.
  * Criação física da estrutura base das 10 páginas obrigatórias do aplicativo, com módulos e configuração do `app-routing.module.ts` suportando parâmetros e rotas principais.
  * Implementação da **Tela de Login**, aplicando padrão visual (cabeçalho, logo, card, inputs), validação simples baseada em `ngModel` com feedback via `ion-toast` e navegação funcional para o menu principal.
  * Implementação da **Tela de Menu Principal** (Dashboard), com interface organizada em grid contendo ícones nativos e links de roteamento para todas as áreas (Chamados, Técnicos, Resumo e Sobre).
  * Implementação da tela **Cadastro de Técnico** com formulários validados (`ngModel`), integração unificada ao array em memória no `ChamadosService` e notificações dinâmicas via `ion-toast`.
  * Implementação da tela **Lista de Técnicos** renderizando dinamicamente os cards a partir da memória (`get tecnicos()` vinculado ao Service), uso de `ion-badge` para situação, tela de estado vazio e deleção de dados via `ion-alert`.
  * Implementação da tela **Cadastro de Chamado**, com múltiplos formulários complexos (`ion-select`, `ion-textarea`, `ion-input` de datas), validações lógicas vinculadas ao `ChamadosService` e seleção dinâmica atrelada à sub-lista de Técnicos Ativos.
  * Implementação da tela **Lista de Chamados**, consumindo dados dinâmicos da memória, aplicando badges coloridos para os status e prioridades, botão para navegação a Detalhes, além da função de exclusão com interceptação de clique nativo.
  * Implementação da tela **Detalhes do Chamado**, consumindo o ID via parâmetro de rota (`ActivatedRoute`), exibindo os dados de forma legível em blocos visuais distintos, com estado vazio (not-found) e botão para navegar para `atualizar-status`.
  * Implementação da tela **Atualizar Status**, resgatando o chamado atual pelo ID, permitindo manipular seu status e notas via Service (`atualizarStatus`) e refletindo a mudança no singleton em memória.
  * Implementação da tela **Resumo dos Chamados**, consumindo dados analíticos on-the-fly (`calcularResumoChamados` e prioridades derivadas no getter) montando um dashboard numérico com badges.
  * Implementação da tela estática **Sobre**, concluindo com isso o desenvolvimento estrutural, visual e lógico das 10 páginas requeridas no projeto acadêmico de Eduardo Henrryk Simonato.
  * **Revisão Visual Geral:** Padronização final de todas as telas (botões, cards de formulários, ícones de estado vazio com `opacity: 0.6` e botões de empty state utilizando `<ion-button fill="outline" shape="round">`) garantindo uma consistência visual premium por todo o app sem alterar qualquer regra de negócio ou lógica da API em memória.
  * **Revisão Funcional e Testes de Fluxo:** Foi executada uma ampla varredura mental e análise de código nos 15 fluxos estabelecidos no escopo. Validado o perfeito funcionamento de getters reativos (`get chamados()`), passagem de IDs por rota (`ActivatedRoute`), transição de formulários vazios e exclusões síncronas. Nenhum *bug* bloqueante ou erro de arquitetura foi encontrado, certificando o app para apresentação.
  * **Correção Definitiva do Login:** Refatoração da tela de login para encapsular o HTML em uma tag `<form (ngSubmit)="entrar()">`, forçando a detecção nativa do `ngModel`. O botão recebeu formatação de cor explícita. O método do TypeScript teve a limpeza prematura de campos removida e a navegação foi garantida via `router.navigateByUrl` com substituição de histórico.
  * **Correção de Compatibilidade Windows (09/05/2026):** Instalado `cross-env` como devDependency e atualizado os scripts `start`, `build`, `ionic:serve` e `ionic:build` no `package.json` para usar `cross-env NODE_OPTIONS=--openssl-legacy-provider ng serve/build`. **Problema original:** a sintaxe `VAR=valor comando` é exclusiva de shells Unix e falhava no Windows com `'NODE_OPTIONS' não é reconhecido como um comando interno`. **Problema secundário:** ao remover a flag sem `cross-env`, o webpack do Angular 12 lançava `ERR_OSSL_EVP_UNSUPPORTED` pois depende de hashing MD4 legado incompatível com o OpenSSL 3 do Node 22.12.0. A solução `cross-env` resolve ambos os problemas de forma multiplataforma. Nenhuma lógica, tela ou service foi alterado.
  * **Correção da Navegação do Menu Principal (09/05/2026):** O arquivo `src/global.scss` estava sem os imports obrigatórios do CSS do Ionic (`@ionic/angular/css/core.css`, `normalize.css`, `structure.css`, `typography.css`, `display.css` e utilitários). Sem esses estilos, o `<ion-router-outlet>` não conseguia gerenciar a pilha de views — a URL mudava ao clicar nos cards, mas a tela do Menu Principal permanecia visível. Adicionados todos os 10 imports de CSS do Ionic conforme padrão de qualquer projeto Ionic starter. Todas as 7 opções do menu agora navegam corretamente. Nenhuma tela, service, model ou regra de negócio foi alterada.
  * **Ajuste Visual nos ion-select (09/05/2026):** Padronização visual dos componentes `<ion-select>` em todo o aplicativo (Telas de Cadastro de Chamado, Cadastro de Técnico e Atualizar Status). Adicionada a propriedade `interface="action-sheet"` para garantir uma exibição mobile moderna com as opções subindo a partir da base da tela, e `cancelText="Cancelar"` para separar visualmente o botão de cancelamento das opções selecionáveis, melhorando a usabilidade e o layout. Nenhuma regra de negócio ou lógica de formatação de dados foi modificada.
  * **Implementação de Atalhos de Teclado (09/05/2026):** Adicionado suporte para navegação e submissão via teclado (ENTER e ESC) utilizando `@HostListener` nos componentes. A tecla ENTER aciona ações principais como "Entrar" (Login) ou "Salvar" (Cadastros e Atualização), prevenindo o acionamento indevido dentro de `<textarea>`. A tecla ESC permite retornar à tela anterior ou ao Menu Principal de forma rápida a partir de telas internas, sendo inteligentemente interrompida se houver um modal nativo do Ionic aberto (`ion-alert`, `ion-select`, etc.). O botão ESC foi intencionalmente ignorado nas telas de Login e Menu Principal para garantir a segurança da sessão e forçar a saída explícita pelo botão "Sair".
  * **Atualização Tela Sobre (09/05/2026):** Adicionado `Antigravity` na lista de tecnologias utilizadas na tela Sobre o Aplicativo.
  * **Login e Controle de Sessão (09/05/2026):** Implementado `SessaoService` para armazenamento em memória do perfil logado (usuario, tecnico, administrador). Ajustada a tela de Login para aceitar apenas os perfis previstos no escopo. Adicionado `AuthGuard` no roteamento do Angular para impedir acessos diretos via URL em páginas não autorizadas, redirecionando com mensagens visuais via Toast. Atualizada a tela de Menu Principal para esconder/mostrar as opções ativamente baseada no tipo de perfil de usuário.
